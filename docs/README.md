# docs/

Project-level documentation. Read in this order:

1. **`project.md`** — architecture, tech-stack decisions, API research, dataset strategy, build plan. The reference doc for "how is this project set up and why."
2. **`gst_invoice_platform_PRD.md`** — the frozen PRD (what + why + eventual business).
3. **`research.md`** — per-topic research (OCR, datasets, validation rules) with source links.
4. **`business_plan.md`** — two-persona onboarding platform design for the `business_layer/`.
5. **`progress.md`** — dated decision and milestone log (append-only).
6. **`learning.md`** — running notes on what we've learned during implementation.

Component-level READMEs (e.g. `extraction_layer/components/ocr/README.md`) stay next to their code — they're close-range docs for a specific module, not project-level narrative.
